# Adding MSI GS66 (SteelSeries KLC, `1038:113a`) to OpenRGB — in-tree guide

This is a **hand-author-it-yourself** guide for submitting GS66 per-key RGB support to OpenRGB by extending
the existing `Controllers/MSILaptopController/`. It exists because **OpenRGB does not accept AI-generated or
AI-co-authored contributions** (`CONTRIBUTING.md`) — so this repo ships the *facts and the exact diff shape*,
and you write/commit the change under your own name (no `Co-Authored-By` trailer).

Everything here was verified against the OpenRGB tree; the alternative that lives in *our* tree (needs no
upstream MR) is the out-of-tree plugin under `openrgb-plugin/`.

## Why this is small

OpenRGB **already implements the KLC protocol.** `Controllers/MSILaptopController/MSILaptopController.cpp`
builds the same 525-byte feature report we reverse-engineered and even comments *"Verified on a GS66
(1038:113A)"*. It just never **registers** our device: detection is **DMI-gated to one Raider laptop** and
bound to PIDs `0x1122/0x1161`, not `0x113a`. So this is a **data + detector** change in **one file**:
`Controllers/MSILaptopController/MSILaptopControllerDetect.cpp`.

## The change (one file)

1. **PID define** near the existing ones (`…Detect.cpp:18-20`):
   ```c
   #define STEELSERIES_MSI_GS66_KLC_PID   0x113A
   ```
2. **GS66 LED + matrix arrays** — paste `openrgb-gs66-arrays.h` from this directory (generated by
   `gen-openrgb-gs66.py` from our hardware-validated `gs66-keymap.json`). It defines
   `msi_gs66_klc_leds[]` (88 keys, no numpad), `MSI_GS66_KLC_MATRIX_HEIGHT/WIDTH`, and
   `msi_gs66_klc_matrix_map[H][W]`. The HID codes are identical to OpenRGB's existing
   `msi_raider_a18_klc_leds`, and the names use the same `KEY_EN_*` constants (`RGBControllerKeyNames.h`),
   so it drops in.
3. **Model table entry** — append to `msi_laptop_models[]` (`…Detect.cpp:163-181`) with your unit's
   **exact** DMI strings:
   ```c
   {
       "Micro-Star International Co., Ltd.",   // /sys/devices/virtual/dmi/id/sys_vendor  (verbatim)
       "GS66 Stealth 12UHS",                   // /sys/devices/virtual/dmi/id/product_name (verbatim)
       msi_gs66_klc_leds,
       MSI_LAPTOP_ARRAY_SIZE(msi_gs66_klc_leds),
       MSI_GS66_KLC_MATRIX_HEIGHT,
       MSI_GS66_KLC_MATRIX_WIDTH,
       (const unsigned int*)msi_gs66_klc_matrix_map,
       NULL, 0, 0,                             // GS66 is keyboard-only (no ALC lightbar)
   },
   ```
4. **PID branch** in `DetectMSILaptop()` (`…Detect.cpp:209`):
   ```c
   else if(info->product_id == STEELSERIES_MSI_GS66_KLC_PID)
   {
       device_type = MSI_LAPTOP_KLC;
   }
   ```
5. **Detector registration** (`…Detect.cpp:233`):
   ```c
   REGISTER_HID_DETECTOR_IPU("MSI GS66 Keyboard", DetectMSILaptop, STEELSERIES_VID, STEELSERIES_MSI_GS66_KLC_PID, 0, 0xFFC0, 0x0001);
   ```
   **Usage is `0x0001`, not `0xF1`** — verified on-device: `hid_enumerate` reports the LED interface as
   `iface=0 usage_page=0xFFC0 usage=0x0001` (the top-level collection usage). `0xF1` is the *output-report*
   usage from the report descriptor and does **not** match what enumeration returns; using it detects nothing.

`MSILaptopController.{h,cpp}`, `RGBController_MSILaptop.{h,cpp}`, `OpenRGB.pro`, and the udev rules are
**generic / auto-generated** and need no edits (build auto-globs `Controllers/*.cpp`; udev is generated by
`scripts/build-udev-rules.sh` from the detector + the `@detectors DetectMSILaptop` doc block).

## Gotchas (each has bitten someone)

- **DMI exact-match, or nothing registers.** `GetMSILaptopModelDMI()` (`…Detect.cpp:183-197`) early-returns
  unless `sys_vendor` **and** `product_name` match exactly. MSI ships many GS66 SKUs with distinct product
  names (`GS66 Stealth 10SGS`, `11UH`, `12UHS`, …). Read yours with
  `cat /sys/devices/virtual/dmi/id/{sys_vendor,product_name}` and paste **verbatim**. To cover multiple SKUs,
  add one row per product_name — they can reuse the same `msi_gs66_klc_leds[]`/matrix pointers.
- **Usage-page filter is conditional.** The `usage_page`/`usage` args of `REGISTER_HID_DETECTOR_IPU` are only
  honored when `USE_HID_USAGE` is defined (`ResourceManager.cpp:86-95`), which on Linux requires
  hidapi-hidraw ≥ 0.10.1 (`OpenRGB.pro:507-517`). The **interface** arg is always honored. Confirm the real
  interface/usage of the LED endpoint from OpenRGB's own log (`--loglevel 6`, line emitted at
  `ResourceManager.cpp:1449`) and adjust; if you can't rely on `USE_HID_USAGE`, fall back to
  `REGISTER_HID_DETECTOR_I(..., 0x113A, <iface>)` pinning just the interface so you don't grab the
  boot-keyboard node.
- **LED count.** Our extraction yields 88 physical keys; the existing `MSILaptopController.cpp` comment says
  the GS66 has 106 LEDs. `SetLEDs` sends `buf[3] = led_count` and handles any count, so this only affects how
  many `leds[]` you declare — keep `klc_leds_count == ARRAY_SIZE(msi_gs66_klc_leds)`. If your GS66 has
  extra non-key zones, enumerate them on-device and extend `msi_gs66_klc_leds[]`; the 88-key set is the
  validated per-key baseline.
- **Matrix positions are best-effort.** `openrgb-gs66-arrays.h`'s `matrix_map` is a US-ANSI draft; verify the
  on-screen grid in OpenRGB against the real keyboard and nudge cells. This never affects per-key control
  (that's driven by `leds[].value` = HID code), only the visual layout.

## Optional: reactive + onboard breathe/wave as device modes

The MVP declares only the **Direct** mode (`MODE_FLAG_HAS_PER_LED_COLOR`, `MODE_COLORS_PER_LED`), which gives
you breathing/wave/rainbow **for free** from OpenRGB's software effect engine. To expose the *firmware's* own
reactive/breathe/wave (battery/standalone use), add extra `mode` entries and dispatch on `active_mode` — the
pattern is `Controllers/SteelSeriesController/SteelSeriesApex3Controller/RGBController_SteelSeriesApex3.cpp:36-64,121-131`
(non-Direct modes use `MODE_COLORS_MODE_SPECIFIC` + `MODE_FLAG_HAS_SPEED/DIRECTION/BRIGHTNESS`; `DeviceUpdateMode()`
passes `mode.value`/`brightness`/`speed` to the controller). Our effect byte-formats are documented in
`KLC-PROTOCOL.md` (cmd `0x0B` program + `0x0E` per-key). Do this as a **separate follow-up MR**, not the first one.

Per-model **color-scale correction** (our `color_scale`, `msi-klc/data/msi-klc-models.json`) would go as a pure
R/G/B transform just before the `buf[offset+1..3]` writes in `SetLEDs` — also a follow-up, since OpenRGB
otherwise sends raw RGB.

## Process (OpenRGB CONTRIBUTING.md)

- Fork on **GitLab** (`gitlab.com/CalcProgrammer1/OpenRGB`); MR source branch must **not** be `master`.
- **Linear history:** `git rebase` onto master, never `git merge`; squash to a tidy set of commits.
- Open the MR titled **`Draft:`** until you've tested on the real GS66; clear `Draft:` only when it works.
- **One functional area per MR** — GS66 device support only, no drive-by cleanup or KLM migration.
- Code style: 4-space soft tabs; braces on their own lines; **uppercase** hex in code (lowercase only in the
  generated udev rules); no space after `if`/`for`; `snake_case` vars, `CamelCase` funcs; prefer `#define`
  over `const`; C-style casts/loops; no `auto`; log via `LogManager`, never `printf`. New files get the SPDX
  `GPL-2.0-or-later` header box. The existing MSILaptop files are a conforming template — copy their style.
- **AI policy:** author and commit this yourself; do **not** add AI author/co-author trailers.

## Direction note (so you follow current conventions)

The project is migrating keymaps to a shared `KeyboardLayoutManager/` (used by ~21 controllers), but
`MSILaptopController` still uses inline per-model `leds[]`/`matrix_map[]` arrays. **Match the inline pattern**
for the GS66 MR. A KLM migration of MSILaptop is a reasonable *separate* proposal later (note: KLC uses
hardware keycodes as `led.value`, which KLM doesn't carry, so you'd keep a keycode table regardless).

## Verification checklist (before clearing `Draft:`)

1. `sys_vendor`/`product_name` copied verbatim from `/sys/devices/virtual/dmi/id/`.
2. Interface/usage confirmed from `--loglevel 6`; detector macro adjusted; falls back to `_I` if no
   `USE_HID_USAGE`.
3. Device appears in OpenRGB as a keyboard; per-key set/clear works across **all** keys incl. Power/Home/End
   (no truncation on high-index keys — the bug the `buf[3]` count fix addressed).
4. Direct mode + one OpenRGB software effect render correctly; colors are accurate (add color-scale later if
   red reads orange).
5. Visual grid matches the physical keyboard (tune `matrix_map` if not).
